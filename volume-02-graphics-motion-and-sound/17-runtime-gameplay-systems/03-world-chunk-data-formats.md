## 17.3 世界分块数据格式

正如我们所见，一个世界分块通常同时包含**静态**和**动态**世界元素。静态几何体可以由一个大型三角形网格表示，也可以由许多较小的网格组成。每个网格都可能被多次**实例化**（instanced）——例如，同一个门的网格可以被该分块中所有门口复用。静态数据通常包括碰撞信息，这些信息可以存储为三角形汤（triangle soup）、一组凸形状，和/或其他更简单的几何形状，例如平面、盒体、胶囊体或球体。其他静态元素包括体积**区域**（regions），它们可用于检测事件或划定游戏世界中的区域；AI **导航网格**（navigation mesh）；一组线段，用于标出背景几何体中可被玩家角色抓取的**边缘**（edges），等等。这里不会深入讨论这些数据格式的细节，因为我们已经在前文各节中讨论过其中大多数内容。

世界分块的动态部分包含该分块中游戏对象的某种表示。一个游戏对象由它的**属性**（attributes）和**行为**（behaviors）定义，而对象的行为由其**类型**（type）直接或间接决定。在以对象为中心的设计中，对象的类型直接决定了运行时需要实例化哪些类来表示该对象。在以属性为中心的设计中，游戏对象的行为由其属性行为的组合决定，但类型仍然决定了该对象应该具有哪些属性（或者也可以说，一个对象的属性定义了它的类型）。因此，对于每个游戏对象，世界分块数据文件通常包含：

- **对象属性的初始值**。世界分块定义了每个游戏对象在首次生成到游戏世界中时应该具有的状态。对象的属性数据可以用多种不同格式存储。下面将考察几种常见格式。
- **某种对象类型说明**。在以对象为中心的引擎中，这可以是一个字符串、一个哈希字符串 ID，或某种其他唯一类型 ID。在以属性为中心的设计中，类型可以被显式存储，也可以由对象所包含的属性集合隐式定义。

### 17.3.1 二进制对象映像

将一组游戏对象存储到磁盘文件中的一种方法，是把每个对象在运行时内存中的样子完整写成二进制映像并存入文件。这使得生成游戏对象变得非常简单。一旦游戏世界分块被加载到内存中，我们就拥有了所有对象的现成映像，因此只需让它们运行起来即可。

当然，事情并不完全如此简单。存储“活的”C++ 类实例的二进制映像会带来许多问题，包括需要以特殊方式处理**指针**（pointers）和**虚函数表**（virtual tables），以及可能需要对每个类实例内部的数据进行**字节序交换**（endian-swap）。（这些技术在 Section 7.2.2.9 中有详细描述。）此外，二进制对象映像不够灵活，也不能很好地应对变更。玩法是任何游戏项目中最动态、最不稳定的方面之一，因此明智的做法是选择一种支持快速开发、并能稳健应对频繁变化的数据格式。因此，二进制对象映像格式通常并不是存储游戏对象数据的好选择（尽管这种格式可能适合更稳定的数据结构，例如网格数据或碰撞几何体）。

### 17.3.2 序列化的游戏对象描述

**序列化**（serialization）是另一种将游戏对象内部状态表示存储到磁盘文件中的方式。与二进制对象映像技术相比，这种方法通常更可移植，也更容易实现。将对象序列化到磁盘时，会要求该对象生成一段数据流，其中包含足够细节，使原始对象之后能够被重建。当对象从磁盘反序列化回内存时，会创建适当类的一个实例，然后读取属性数据流，以初始化新对象的内部状态。如果原始序列化数据流是完整的，那么从各种实际目的来看，新对象应该与原始对象完全一致。

某些编程语言原生支持序列化。例如，C# 和 Java 都提供了标准化机制，用于将对象实例序列化到 XML 文本格式，并从 XML 文本格式反序列化回来。遗憾的是，C++ 语言并不提供标准化序列化设施。不过，在游戏行业内外，已经有许多 C++ 序列化系统被成功构建出来。这里不会讨论如何编写 C++ 对象序列化系统的所有细节，但会描述这种数据格式，以及为了让序列化在 C++ 中工作而需要编写的几个主要系统。

序列化数据并不是对象的二进制映像。相反，它通常会以一种更方便、更可移植的格式存储。XML 是对象序列化的一种常见格式，因为它得到广泛支持并且已经标准化，具有一定的人类可读性，并且对层级数据结构有很好的支持；而在序列化相互关联的游戏对象集合时，层级数据结构经常会出现。遗憾的是，XML 的解析速度出了名地慢，这可能会增加世界分块加载时间。出于这个原因，一些游戏引擎会使用专有二进制格式，它比 XML 文本解析更快，也更紧凑。

许多游戏引擎（以及非游戏对象序列化系统）已经转向基于文本的 JSON 数据格式 [389]，将其作为 XML 的替代方案。JSON 也广泛用于万维网上的数据通信。例如，Facebook API 完全使用 JSON 进行通信。

对象与磁盘之间进行序列化和反序列化的机制，通常通过以下两种基本方式之一实现：

- 可以在基类中引入一对虚函数，例如 `SerializeOut()` 和 `SerializeIn()`，并安排每个派生类为它们提供自定义实现，使这些实现“知道”如何序列化该特定类的属性。
- 可以为 C++ 类实现一个**反射系统**（reflection system）。然后，我们可以编写一个泛型系统，自动序列化任何拥有可用反射信息的 C++ 对象。

**反射**（reflection）是 C# 等语言中使用的术语。简言之，反射数据是对类内容的运行时描述。它存储的信息包括：类名、类中包含哪些数据成员、每个数据成员的类型、每个成员在对象内存映像中的偏移量，并且还包含该类所有成员函数的信息。给定任意 C++ 类的反射信息，我们就可以相当容易地编写一个通用对象序列化系统。

C++ 反射系统中棘手的部分，是为所有相关类生成反射数据。这可以通过多种方式完成：将类的数据成员封装到 `#define` 宏中，以提取相关反射信息；提供一个可由每个派生类重写的虚函数，使其返回该类适当的反射数据；为每个类手写一个反射数据结构；或者采用其他有创造性的方式。

除了属性信息之外，序列化数据流总会包含每个对象的**类**或**类型**的名称或唯一 ID。类 ID 用于在对象从磁盘序列化到内存中时实例化适当的类。类 ID 可以存储为字符串、哈希字符串 ID，或其他某种唯一 ID。

遗憾的是，C++ 并不提供一种只根据类名（字符串或 ID）来实例化类的方式。类名必须在编译期已知，因此必须由程序员硬编码（例如 `new ConcreteClass`）。为了绕过语言的这一限制，C++ 对象序列化系统总会包含某种形式的**类工厂**（class factory）。工厂可以通过任意数量的方式实现，但最简单的方法是创建一个数据表，将每个类名/ID 映射到某种函数或函数对象，这个函数或函数对象已经被硬编码为可以实例化该特定类。给定类名或 ID 后，我们只需在表中查找对应函数或函数对象，并调用它来实例化该类。

### 17.3.3 生成器与类型模式

二进制对象映像和序列化格式都有一个阿喀琉斯之踵。它们都由所存储游戏对象类型的运行时实现来定义，因此都要求世界编辑器深入了解游戏引擎的运行时实现。例如，为了让世界编辑器写出一组异构游戏对象的二进制映像，它要么必须直接链接到运行时游戏引擎代码，要么必须 painstakingly 手写生成一些字节块，使其精确匹配游戏对象在运行时的数据布局。序列化数据与游戏对象实现的耦合程度较低，但同样，世界编辑器要么需要与运行时游戏对象代码链接，以便访问各类的 `SerializeIn()` 和 `SerializeOut()` 函数；要么需要以某种方式访问这些类的反射信息。

可以通过以与实现无关的方式抽象游戏对象描述，来打破游戏世界编辑器与运行时引擎代码之间的耦合。对于世界分块数据文件中的每个游戏对象，我们都存储一个小数据块，通常称为**生成器**（spawner）。生成器是游戏对象的一种轻量级、纯数据表示，可用于在运行时实例化并初始化该游戏对象。它包含游戏对象工具侧**类型**（tool-side type）的 ID。它还包含一个由简单键值对组成的表，用于描述游戏对象的初始属性。这些属性通常包括模型到世界变换，因为大多数游戏对象在世界空间中都有独立的位置、朝向和缩放。当游戏对象被生成时，会根据生成器的类型实例化适当的一个或多个类。然后，这些运行时对象可以查询键值对字典，以适当地初始化它们的数据成员。

生成器可以被配置为在加载后立即生成其游戏对象，也可以保持休眠，直到游戏过程中稍后某个时间被要求生成。生成器可以作为一等对象实现，因此它们可以拥有方便的功能接口，并且除了对象属性之外，还可以存储有用的元数据。生成器甚至可以用于生成游戏对象以外的目的。例如，在 Naughty Dog 引擎中，设计师使用生成器来定义游戏世界中的重要点或坐标轴。这些生成器称为**位置生成器**（position spawners）或**定位器生成器**（locator spawners）。定位器在游戏中有许多用途，例如：

- 为 AI 角色定义兴趣点；
- 定义一组坐标轴，使一组动画能够相对于这些坐标轴以完美同步方式播放；
- 定义粒子效果或音频效果应该产生的位置；
- 定义沿赛道分布的路径点；

等等。

#### 17.3.3.1 对象类型模式

游戏对象的属性和行为由其类型定义。在采用生成器式设计的游戏世界编辑器中，游戏对象类型可以由数据驱动的**模式**（schema）表示；该模式定义了在创建或编辑该类型对象时，用户应该能够看到的一组属性。在运行时，工具侧对象类型可以通过硬编码或数据驱动的方式，映射到必须实例化的一个类或一组类，从而生成给定类型的游戏对象。

类型模式可以存储在一个简单文本文件中，供世界编辑器使用，也供用户检查和编辑。例如，一个模式文件可能大致如下：

    enum LightType
    {
        Ambient, Directional, Point, Spot
    }

    type Light
    {
        String      UniqueId;
        LightType   Type;
        Vector      Pos;
        Quaternion  Rot;
        Float       Intensity : min(0.0), max(1.0);
        ColorARGB   DiffuseColor;
        ColorARGB   SpecularColor;
        // ...
    }

    type Vehicle
    {
        String         UniqueId;
        Vector         Pos;
        Quaternion     Rot;
        MeshReference  Mesh;
        Int            NumWheels : min(2), max(4);
        Float          TurnRadius;
        Float          TopSpeed : min(0.0);
        // ...
    }

    //...

上面的示例揭示了一些重要细节。你会注意到，除了名称之外，每个属性的**数据类型**也被定义了。它们可以是简单类型，例如字符串、整数和浮点值；也可以是专门类型，例如向量、四元数、ARGB 颜色，或对特殊资源类型的引用，例如网格、碰撞数据等。在这个示例中，我们甚至提供了定义枚举类型的机制，例如 `LightType`。另一个细微之处在于，对象类型模式会向世界编辑器提供额外信息，例如编辑该属性时应该使用何种 GUI 元素。有时，属性的 GUI 需求由其数据类型隐含决定——字符串通常使用文本字段编辑，布尔值通过复选框编辑，向量则通过三个文本字段编辑 x、y 和 z 坐标，或者通过一个专门用于在 3D 中操纵向量的 GUI 元素编辑。模式还可以指定供 GUI 使用的元信息，例如整数和浮点属性允许的最小值和最大值、下拉组合框可选项列表，等等。

有些游戏引擎允许对象类型模式像类一样继承。例如，每个游戏对象都需要知道自己的类型，并且必须拥有一个**唯一 ID**，以便在运行时与所有其他游戏对象区分开来。这些属性可以在顶层模式中指定，所有其他模式都从该模式派生。

#### 17.3.3.2 默认属性值

可以想象，典型游戏对象模式中的属性数量可能会变得相当庞大。这意味着，对于游戏设计师放入游戏世界中的每一种游戏对象类型的每一个实例，都必须指定大量数据。在模式中为许多属性定义**默认值**会非常有用。这允许游戏设计师以很少工作量放置某个游戏对象类型的“普通”实例，但仍然允许他们根据需要对特定实例上的属性值进行细调。

当某个特定属性的默认值发生变化时，默认值会带来一个固有问题。例如，我们的游戏设计师最初可能希望 Orc 拥有 20 点生命值。在制作数月之后，设计师可能决定 Orc 应该更强大，默认拥有 30 点生命值。现在，任何新放入游戏世界的 Orc 都会拥有 30 点生命值，除非另有指定。但在这一变更之前已经放入游戏世界分块中的所有 Orc 怎么办？是否需要找到所有这些先前创建的 Orc，并手动将它们的生命值改为 30？

理想情况下，我们希望将生成器系统设计成：默认值的变化会自动传播到所有尚未显式覆盖默认值的既有实例。实现这一功能的一种简单方法，是直接省略那些值与默认值没有差异的属性键值对。每当生成器中缺少某个属性时，就可以使用适当的默认值。（这假定游戏引擎可以访问对象类型模式文件，从而读取属性默认值。或者也可以由工具完成这件事——在这种情况下，传播新的默认值只需要简单重建所有受该变化影响的世界分块。）在我们的例子中，大多数既有 Orc 生成器根本不会拥有 `HitPoints` 键值对（当然，除非某个生成器的生命值已经被手动改成不同于默认值的值）。因此，当默认值从 20 变为 30 时，这些 Orc 会自动使用新值。

有些引擎允许在派生对象类型中覆盖默认值。例如，一个名为 `Vehicle` 的类型模式可能定义默认 `TopSpeed` 为每小时 80 英里。派生的 `Motorcycle` 类型模式可以将这个 `TopSpeed` 覆盖为每小时 100 英里。

#### 17.3.3.3 生成器与类型模式的一些收益

将生成器与游戏对象实现分离，其关键收益是**简单性**（simplicity）、**灵活性**（flexibility）和**稳健性**（robustness）。从数据管理角度来看，处理一张键值对表，要比管理带有指针修正的二进制对象映像或自定义序列化对象格式简单得多。键值对方法还使数据格式极其灵活，并且对变更非常稳健。如果游戏对象遇到不期望看到的键值对，它可以直接忽略它们。同样，如果游戏对象找不到自己需要的某个键值对，它也可以选择改用默认值。这使键值对数据格式能够很好地应对设计师和程序员双方所做的变更。

生成器还简化了游戏世界编辑器的设计和实现，因为编辑器只需要知道如何管理键值对列表和对象类型模式。它不需要以任何方式与运行时游戏引擎共享代码，并且只与引擎的实现细节保持非常松散的耦合。

生成器和原型（archetypes）为游戏设计师和程序员提供了大量灵活性与能力。设计师可以在世界编辑器中定义新的游戏对象类型模式，几乎不需要程序员介入。程序员可以在日程允许时再实现这些新对象类型的运行时实现。程序员不需要在每个新对象类型被添加时立即提供其实现，以避免游戏崩溃。新的对象数据可以安全地存在于世界分块文件中，无论是否已经有对应的运行时实现；运行时实现也可以存在，而不一定要求世界分块文件中已经有对应数据。
