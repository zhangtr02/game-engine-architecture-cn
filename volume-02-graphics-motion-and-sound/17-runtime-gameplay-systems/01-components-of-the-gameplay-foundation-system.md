## 17.1 玩法基础系统的组成部分

大多数游戏引擎都会提供一套运行时软件组件，它们共同构成一个框架，使一款游戏独有的规则、目标以及动态世界元素能够在其上构建。在游戏行业中，这些组件并没有一个标准名称，但在本书中，我们将把它们统称为引擎的**玩法基础系统**（gameplay foundation system）。如果可以在游戏引擎和游戏本身之间合理地画出一条界线，那么这些系统就位于这条界线的正下方。理论上，人们可以构建出在很大程度上与具体游戏无关的玩法基础系统。然而在实践中，这些系统几乎总会包含特定类型或特定游戏的细节。事实上，引擎与游戏之间的界线也许最好被想象成一大片模糊区域——一个跨越这些组件的渐变带，它将引擎与游戏连接起来。在某些游戏引擎中，人们甚至可以认为玩法基础系统完全位于引擎—游戏界线之上。当涉及玩法组件的设计与实现时，不同游戏引擎之间的差异最为明显。话虽如此，不同引擎之间仍然存在数量惊人的共同模式，而这些共性正是本章讨论的主题。

每个游戏引擎处理玩法软件设计问题的方式都略有不同。不过，大多数引擎都会以某种形式提供以下主要子系统：

- **运行时游戏对象模型**（runtime game object model）。这是通过世界编辑器向游戏设计师展示的抽象游戏对象模型的一种实现。
- **关卡管理与流式加载**（level management and streaming）。该系统负责加载和卸载玩法发生所在虚拟世界的内容。在许多引擎中，关卡数据会在游戏过程中流式载入内存，从而制造出大型无缝世界的错觉（尽管实际上它被拆分成了离散分块）。
- **实时对象模型更新**（real-time object model updating）。为了让世界中的游戏对象能够自主行动，每个对象都必须周期性更新。正是在这里，游戏引擎中的各种彼此分散的系统真正汇聚为一个有凝聚力的整体。
- **消息传递与事件处理**（messaging and event handling）。大多数游戏对象都需要彼此通信。这通常通过一个抽象消息系统完成。对象之间的消息常常表示游戏世界状态的变化，这种变化称为**事件**（events）。因此，在许多工作室中，消息系统也被称为**事件系统**（event system）。
- **脚本**（scripting）。用 C 或 C++ 这样的语言编写高层游戏逻辑可能很繁琐。为了提高生产效率、支持快速迭代，并将更多能力交到团队中非程序员手中，游戏引擎通常会集成一种脚本语言。这种语言可以是基于文本的，例如 Python 或 Lua；也可以是图形化语言，例如 Unreal 的 Blueprints。
- **目标与游戏流程管理**（objectives and game flow management）。该子系统管理玩家目标以及游戏的整体流程。它通常由玩家目标的一个序列、树或广义图来描述。目标通常会被组织成章节，尤其是在许多现代游戏这种高度叙事驱动的游戏中。游戏流程管理系统会管理游戏的整体流程，追踪玩家对目标的完成情况，并在目标完成后，将玩家从游戏世界的一个区域引导到下一个区域。有些设计师将其称为游戏的“脊柱”（spine）。

在这些主要系统中，运行时对象模型大概是最复杂的。它通常会提供以下大部分功能，甚至全部功能：

- **动态生成与销毁游戏对象**（spawning and destroying game objects dynamically）。游戏世界中的动态元素常常需要在玩法过程中出现和消失。医疗包在被拾取后会消失，爆炸会出现然后消散，敌方增援会在你以为已经清空关卡时神秘地从拐角处冒出来。许多游戏引擎会提供一个系统，用于管理与动态生成的游戏对象相关的内存和其他资源。另一些引擎则干脆完全不允许动态创建或销毁游戏对象。
- **与底层引擎系统的链接**（linkage to low-level engine systems）。每个游戏对象都会以某种方式链接到一个或多个底层引擎系统。大多数游戏对象在视觉上由可渲染三角形网格表示。有些对象带有粒子效果。许多对象会发出声音。有些对象会播放动画。许多对象具有碰撞，有些对象则由物理引擎进行动态模拟。玩法基础系统的一项主要职责，就是确保每个游戏对象都能够访问其所依赖的引擎系统服务。
- **对象行为的实时模拟**（real-time simulation of object behaviors）。从本质上说，游戏引擎是一个基于代理模型的实时动态计算机仿真系统。这只是一种更正式的说法，意思是游戏引擎需要随时间动态更新所有游戏对象的状态。对象可能需要按照非常特定的顺序更新，这一顺序部分由对象之间的依赖关系决定，部分由对象对各种引擎子系统的依赖关系决定，也部分由这些引擎子系统彼此之间的相互依赖关系决定。
- **定义新游戏对象类型的能力**（ability to define new game object types）。随着游戏开发进行，每款游戏的需求都会不断变化和演化。因此，游戏对象模型必须足够灵活，以便能够轻松添加新的对象类型，并将其暴露给世界编辑器。在理想情况下，应该能够以完全数据驱动的方式定义一种新的对象类型。然而，在许多引擎中，添加新的游戏对象类型仍然需要程序员提供服务。
- **唯一对象 ID**（unique object ids）。典型游戏世界包含数百甚至数千个不同类型的独立游戏对象。在运行时，能够识别或搜索某个特定对象非常重要。这意味着每个对象都需要某种唯一标识符。人类可读的名称是最方便的一类 ID，但我们必须警惕运行时使用字符串所带来的性能成本。整数 ID 是最高效的选择，但对于人类游戏开发者来说非常难以使用。可以说，最佳方案是使用哈希字符串 ID（见 [Section 6.4.3.1](../../volume-01-foundations-and-core-engine-systems/06-engine-support-systems/04-strings.md#6431-哈希字符串标识符)）作为对象标识符，因为它们像整数一样高效，同时又可以转换回字符串形式，便于阅读。
- **游戏对象查询**（game object queries）。玩法基础系统必须提供某种手段，用于在游戏世界中查找对象。我们可能想通过唯一 ID 查找某个特定对象，或查找某一特定类型的所有对象；也可能希望根据任意条件执行高级查询（例如，查找玩家角色 20 m 半径范围内的所有敌人）。
- **游戏对象引用**（game object references）。一旦找到了对象，我们就需要某种机制来持有指向它们的**引用**（references），这种引用可以只在单个函数中短暂存在，也可以保持更长时间。对象引用可以简单到只是一个指向 C++ 类实例的指针，也可以是更复杂的形式，例如句柄或引用计数智能指针。
- **有限状态机支持**（finite state machine support）。许多类型的游戏对象最适合建模为有限状态机（finite state machines, FSM）。有些游戏引擎允许一个游戏对象处于多种可能状态之一，并且每种状态都拥有自己的属性和行为特征。
- **网络复制**（network replication）。在联网多人游戏中，多台游戏机器会通过局域网或互联网连接在一起。某个特定游戏对象的状态通常由一台机器拥有和管理。然而，该对象的状态也必须被**复制**（replicated，即通信传递）到参与多人游戏的其他机器上，使所有玩家都能对该对象拥有一致的视图。
- **保存与加载 / 对象持久化**（saving and loading / object persistence）。许多游戏引擎允许将世界中游戏对象的当前状态保存到磁盘，并在之后重新加载。这可能是为了支持“随处保存”（save anywhere）的存档系统，也可能是实现网络复制的一种方式，或者它也可能只是加载由世界编辑器工具制作的游戏世界分块的主要手段。对象持久化通常需要某些语言特性，例如**运行时类型识别**（runtime type identification, RTTI）、**反射**（reflection）和**抽象构造**（abstract construction）。RTTI 和反射为软件提供了在运行时动态确定对象**类型**，以及其类提供了哪些**属性**和**方法**的手段。抽象构造则允许在不必硬编码类名的情况下创建类的实例——当从磁盘将对象实例反序列化到内存中时，这是一项非常有用的功能。如果你选择的语言并不原生支持 RTTI、反射和抽象构造，也可以手动添加这些特性。
