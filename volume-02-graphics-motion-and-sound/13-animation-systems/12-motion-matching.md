## 13.12 运动匹配

**运动匹配**（motion matching, MM）是混合树和扁平加权平均式动画混合的一种替代方案。运动匹配技术可以让动作状态机（ASM）的搭建和维护成本更低，也更不容易出错。它还有潜力生成极其平滑且自然的角色运动。在以下几节中，我们将探讨运动匹配的动机，并看看它通常是如何实现和使用的。

### 13.12.1 动作状态机的问题

尽管动作状态机的概念在原则上相当简单，但在实践中，真实游戏角色的 ASM 很快就会变得非常庞大且高度复杂。任何有经验的动画开发者都会告诉你，构建、修改和维护一个庞大的 ASM 是一项艰巨且容易出错的任务。动画师必须花费大量精力准备 ASM 中的动画片段，使它们满足严格约束，例如匹配核心姿态、确保循环动画之间相位匹配，等等。一旦片段被适当准备好，过渡窗口就必须被细致地标记出来，复杂的过渡图也必须搭建完成。大量有时很反直觉的控制参数必须暴露给引擎的其他部分。然后，还需要有人编写角色控制器逻辑，用这些参数来操纵各种片段的混入和混出，并且必须以非常精确的时机把这一切做得恰到好处，才能让角色运动看起来流畅且符合生物力学规律。

#### 13.12.1.1 实现符合生物力学规律的运动

实现平滑且物理上可信的角色运动本身就是一项复杂而艰巨的任务。例如，为了让奔跑中的角色停下来，我们需要从奔跑循环动画混合到一个“奔跑到待机”的过渡片段，然后再混合到循环待机动画中。为了做到这一点且不产生脚部滑动，我们需要等到奔跑循环的姿态与“奔跑到待机”动画的第一帧相匹配；然后才能开始混合到过渡动画中，同时逐渐降低角色的速度。在“奔跑到待机”片段结束时，当角色移动速度降为零后，我们可以混合到循环待机动画中。要把这一过程编排得看起来完全平滑，既依赖精确定时的逻辑，也依赖以恰当方式进行动作捕捉或手工制作的动画资产，使这些片段能够无缝拼接在一起。

如果我们还需要角色在 3D 世界中的某个特定停止点停下来，情况就会变得更糟，例如角色必须停在梯子底部，然后立即开始攀爬。为此，我们要么接受一定程度的脚部滑动，并“作弊”调整移动速度以命中目标；要么需要预先减慢奔跑循环，使一旦混入停止动画后，目标点能够被命中。

这里我们只讨论了一个具体案例——让一个正在移动的角色停下来。对于我们希望角色执行的几乎任何运动，都存在大量类似问题。待机到移动的过渡与移动到待机的过渡有类似问题。攀爬和翻越要求角色整体位置与其手脚在世界中所攀爬或翻越的障碍物位置之间进行仔细配准。而也许最棘手的挑战之一，是让角色沿曲线路径奔跑。当真实人类在奔跑中转弯时，往往会向转弯方向倾斜，导致他们的脚从质心下方摆出。使用动画混合来复制这种运动确实非常棘手。即使 ASM 已经被 painstakingly 搭好，控制代码也已经写好，角色运动有时仍可能出现跳变，脚仍可能滑动，角色动作仍可能显得僵硬且不符合生物力学规律！

#### 13.12.1.2 逻辑 ASM 与物理 ASM

每个动画开发者都会遇到的另一个问题，是游戏需求（例如对玩家控制的响应性、攻击时机等）与让角色运动看起来流畅且符合生物力学规律之间的根本矛盾。例如，游戏设计师可能希望角色能够在 $T$ 秒内从全速冲刺进入静止待机状态，并且停止时移动距离不超过 $x$ 米。即使我们有一个“奔跑到待机”的过渡片段，其持续时间正好是 $T$ 秒，根运动覆盖的距离也正好是 $x$ 米，我们仍然需要等到奔跑循环的姿态与它的第一帧匹配，才能开始混合进去。这意味着我们很可能会比预期花更长时间才能停下，并错过目标点。而且，更多时候，游戏需求并不会与手头动画的特性完全一致，这意味着我们需要进行时间缩放、滑动，或以其他方式“作弊”来满足游戏需求。对于我们希望角色执行的几乎每一种运动，这类问题都会反复出现。

解决游戏需求与角色生物力学需求之间矛盾的一种方法，是构建两个动作状态机——一个用于玩法机制，另一个用于生成符合生物力学规律的运动。我们将前者称为**逻辑状态机**（logical state machine），将后者称为**物理状态机**（physical state machine）。

逻辑 ASM 通常比其物理对应物简单得多。它通常包含较粗粒度的状态，例如 “Locomoting”“Jumping”“Falling”“Climbing” 和 “Dying”。逻辑 ASM 的每个状态都对应一个复杂得多的物理 ASM。例如，仅在逻辑 “Locomoting” 状态内部，物理 ASM 通常就包含一组复杂的待机、行走、奔跑、转向、扫射，以及各种过渡动画片段。通常，游戏逻辑通过人类玩家提供的 HID 输入，或通过 NPC AI 系统生成的命令来控制逻辑 ASM。逻辑状态再反过来控制物理 ASM，以实现期望的角色运动。物理 ASM 也负责处理逻辑状态之间的混合。

大约十年前，游戏开发者开始问自己：物理 ASM 是否可以用更好的方式构建？他们寻求一种技术，它应该：

- 比手工构建的 ASM 更不费力，也更不容易出错；
- 看起来更符合生物力学规律；
- 同时能够非常快速地响应逻辑 ASM 给出的命令，确保人类玩家感觉自己能够直接控制角色；
- 并且能够满足玩法约束，例如紧密跟随期望路径、准确命中期望停止点，等等。

他们的努力最终促成了一种用于构建物理 ASM 的技术，它很好地满足了上述所有标准。这项技术称为**运动匹配**（motion matching, MM）。

### 13.12.2 运动匹配的先驱

运动匹配最初被开发出来时，并不是一个全新的想法。它代表了早期角色动画技术的一种演进，而这些早期技术都旨在简化构建物理 ASM 的过程，同时提升其流畅性和生物力学正确性。下面几节将讨论运动匹配最重要的两个先驱。

#### 13.12.2.1 运动图

在 SIGGRAPH 2002 上，Lucas Kovar、Michael Gleicher 和 Frédéric Pighin 提出了一种构建物理 ASM 的新方法，称为**运动图**（motion graphs）[305]。基本思想是把游戏中的所有动画片段放入一个数据库，并为每个片段标注到其他片段的过渡。这些过渡通常由源片段中的时间窗口限制。例如，在一个循环向前奔跑片段中，可以定义左脚或右脚着地的时间窗口，并使用这些时间窗口来指定允许过渡到其他动画，且这些其他动画分别以左脚或右脚着地开始。一旦运动图构建完成，就可以通过在运动图中“走”一条路径来驱动物理状态机。

运动图可以让物理 ASM 的搭建更简单，但它们仍然存在响应性问题，因为我们仍然需要等待合适的时间窗口，才能过渡到图中的其他节点。运动图对于复杂性问题也没有太大帮助——它们仍然要求动画团队费力地为数据库中的所有动画标注过渡和时间窗口。

#### 13.12.2.2 运动场

2010 年，Yongjoon Lee、Kevin Wampler、Gilbert Bernstein、Jovan Popović 和 Zoran Popović 发表了一篇题为 “Motion fields for interactive character locomotion” 的论文 [306]。在这篇论文中，作者提出了一种通过沿高维向量场中的连续路径移动来生成角色运动的系统，他们称其为**运动场**（motion field）。运动场中的每个点都表示角色身体的一个唯一配置——即其骨架姿态以及各关节的速度/动量。但运动场并不只是一组 $N$ 维点——它是一个向量场。运动场中的向量描述了角色如何合理地从一个配置移动到另一个配置。这在精神上类似于电磁力场表示带电粒子在三维空间中每个不同点上将会移动的方向。

然而，与简单的电磁场不同，运动场的维度远高于三维，并且它可以在这个 $N$ 维空间中的每个点处包含多个方向向量，对应不同类型的运动。为了让角色四处移动，我们沿着其运动场中的路径前进，并遵循其中包含的向量方向性。要改变角色正在做的事情，就改变穿过运动场的期望路径方向，在场内从一个点移动到另一个点时选择不同的过渡向量。

运动场方法可以产生比运动图方法响应性高得多的动画，因为它不需要等待合适的时间窗口到来才能过渡到另一个动画——我们可以在任意时刻从高维空间中的任何点移动到任何相邻点。然而，运动场很难使用，因为它涉及一些复杂数学，而且并不直观。

### 13.12.3 引入运动匹配

运动匹配基于运动场的核心思想，但它以更直观、更直接、数学负担更轻的方式处理问题。该技术最早由 Ubisoft Toronto 的一组动画师和动画程序员开发，其中包括 Michael Buttner、Mike Wasilewski、Alex Bereznyak、Betina Marquis 和 Kristjan Zadziuk；Kristjan 在 GDC 2016 上介绍了该技术 [307]。该技术首次商业化使用于 Ubisoft Montréal 的游戏 *For Honor*（该游戏也在 GDC 2016 上由动画程序员 Simon Clavet 进行了介绍 [308]），此后又被用于许多游戏，包括 Naughty Dog 的 *The Last of Us Part II*（动画师 Michal Mach 和 Maksym Zhuravlov 在 GDC 2021 上进行了介绍 [309]）。运动匹配技术现在已经可用于 Unreal Engine 5 [310]，并且也以扩展形式可用于 Unity 引擎 [311]。

为了让运动匹配工作，我们首先构建一个大型“扁平”动画片段数据库。对于每个片段的每一帧，我们定义一个相对较小的数据向量，用于描述角色在该帧上的**配置**（configuration）。这些配置信息包括：

- 根关节的局部空间位置和速度；
- 若干其他“关键”关节的局部空间位置和速度，例如脚踝、大腿、手腕和脊柱；
- 角色局部空间原点在短时间窗口内的过去和未来轨迹，例如从过去 1 秒到未来 1 秒；
- 可选标签，用于定义每个动画片段的预期用途，并因此定义该片段何时适合播放。

为了给角色制作动画，我们指定角色当前配置以及期望的未来轨迹。然后在动画数据库中搜索一个片段，以及该片段中的一帧，使其最接近期望条件。例如，假设我们的角色当前正以 $v$ 米/秒的速度径直向前奔跑，右脚刚刚着地，左手向前。同时假设我们希望角色保持以速度 $v$ 奔跑，但开始向右转，沿半径为 $r$ 的圆弧运动。我们可以在动画数据库中搜索一个姿态，它匹配角色当前姿态（右脚着地、左手向前），其速度非常接近 $v$，并且其未来轨迹尽可能接近期望未来轨迹。如果搜索恰好返回当前动画片段，那么我们就继续播放它。如果搜索返回了一个不同片段，那么我们就立即开始混合到新片段中，且混合目标是匹配算法所选中的那一帧。

这一匹配过程会每帧重复执行。角色的期望未来配置和运动来自玩家输入（对于玩家角色而言），或来自 AI 系统指定的期望路径（对于非玩家角色而言）。

运动匹配使我们能够在整个动画数据库中找到任意一个片段，使其匹配角色当前姿态和期望未来运动，并直接混合到最匹配的动画帧——不必等待过渡窗口，也不必手工设置过渡图。因此，运动匹配数据库比传统物理 ASM 更容易搭建。匹配算法在概念上很简单、直观，也相当容易实现。而且，由于该算法既匹配角色当前姿态和关节速度，又选择能够满足未来运动需求的片段，运动匹配通常会产生平滑、响应迅速且符合生物力学规律的角色运动。

### 13.12.4 运动匹配详解

#### 舞蹈卡与动画数据库

运动匹配数据库需要包含大量动画，其中要有丰富多样的角色运动。它通常需要包含我们在手工构建物理 ASM 时会制作的所有同类动画片段——待机、行走、奔跑、扫射、转向，以及大量过渡。如果希望游戏内角色能够行走和奔跑，那么数据库中就需要包含一整套行走时的运动，以及一整套奔跑时的运动。如果希望角色能够疲惫或受伤移动，那么也需要为这些情况准备完整的一组运动。

构建动画数据库的一种方法，是在**动作捕捉**（motion capture，也称 mocap）工作室中捕捉真人演员的动作。演员会被记录下来，执行一套精心构建的运动序列，这称为**舞蹈卡**（dance card）。舞蹈卡的目的，是确保动画数据库事实上至少包含我们希望游戏内角色能够执行的每一种运动的一个示例。舞蹈卡可以作为一条很长的连续运动序列进行动作捕捉，也可以作为许多较短的运动片段进行捕捉。

#### 运动匹配算法

下面给出运动匹配更新函数的伪代码，它基于 Simon Clavet 在其 GDC 演讲中展示的代码 [308]。其思想很简单：首先像平常一样更新当前正在播放的动画片段，并提取当前姿态。接着，遍历数据库中每个动画的每一帧——换句话说，遍历数据库中的每一个姿态。对于这些候选姿态中的每一个，计算其代价。保留代价最低的姿态。最后，如果最佳姿态位于不同于当前播放片段的另一个片段中，就混合到它。

~~~cpp
class MMAnimationController
{
    MMClipDatabase* m_pClipDatabase;
    int m_curClipIndex;
    float m_curClipTime;

public:
    void MMUpdate(Goal goal, float dt)
    {
        m_curClipTime += dt;
        Pose curPose = EvalPose(m_curClipIndex,
                                m_curClipTime);

        float bestCost = FLT_MAX;
        Pose bestPose;

        for (Pose candidatePose : m_poseDatabase)
        {
            float candidateCost = ComputeCost(curPose,
                                              candidatePose,
                                              goal);

            if (candidateCost < bestCost)
            {
                bestCost = candidateCost;
                bestPose = candidatePose;
            }
        }

        if (!IsSameClipAndFrame(curPose, bestPose))
        {
            BlendToClip(bestPose.m_clipIndex,
                        bestPose.m_clipTime);
        }
    }
};
~~~

`ComputeCost()` 函数只是把候选姿态本身的代价与候选姿态在给定目标轨迹下的未来代价组合起来：

~~~cpp
float ComputeCost(Pose curPose,
                  Pose candidatePose,
                  Goal goal,
                  float responsiveness)
{
    cost = ComputeCurrentCost(curPose, candidatePose)
        + responsiveness * ComputeFutureCost(candidatePose, goal);
    return cost;
}
~~~

`Goal` 数据结构可以包含任何我们希望用于匹配的条件。至少，它应该包含一条期望轨迹。数据库中的每个姿态都需要用其过去和未来轨迹进行标注，而这些轨迹可以与目标轨迹使用相同的数据结构：

~~~cpp
struct TrajectoryPoint
{
    Point   m_position;
    Quat    m_orientation;
    float   m_timeOffset; // 正值表示未来点，
                          // 负值表示过去点
};

class Goal
{
    // 任意值，但通常小于 8
    constexpr kTrajCapacity = 4;

    TrajectoryPoint   m_desiredTrajectory[kTrajCapacity];

    // 其他期望条件
    Vector m_desiredVelocity;
    // 等等
};
~~~

### 13.12.5 运动匹配实践

当游戏团队第一次尝试运动匹配时，通常会有两种反应：第一，“哇，它能工作，而且看起来真的很好！” 第二，“嗯，其实这里面有很多问题……” 如果没有仔细调校数据库中的动画、你选择用于匹配的动画特征以及代价函数，运动匹配可能会表现出一系列不理想的行为。

例如，要强迫匹配算法持续播放单个片段，并按照预期那样以单调递增的时间索引播放它，可能会很困难。这是因为匹配算法天然倾向于最终选择我们已经播放过的过去帧，而不是播放我们预期会看到的“下一帧”。通常需要内置一个偏置，人为降低当前播放片段未来帧的代价，以避免这一问题。

运动匹配也不是一个特别对动画师友好的系统。动画师必须把大量动画数据投入一个本质上像黑盒的大系统中，而系统选择要播放的动画，是基于动画的若干特征，例如关节位置、关节速度，以及过去和未来轨迹——这些特征并不容易由动画师可视化或概念化。

对运动匹配的完整讨论远远超出了本书范围。不过，我强烈建议你观看前文引用的 GDC 演讲（这里再次列出：[307]、[308]、[309]），因为这些演讲比本章更深入地讨论了运动匹配，并且非常出色地介绍了在游戏中实现运动匹配时的最佳实践和实际注意事项。
