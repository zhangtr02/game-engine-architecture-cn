## 10.1 日志记录与跟踪

还记得你第一次用 BASIC 或 Pascal 写程序的时候吗？（好吧，也许你不记得。如果你比我年轻很多——而且这种可能性相当大——你第一次写程序时用的可能是 Java，或者也许是 Python 或 Lua。）无论如何，你大概还记得当时是如何调试程序的。你知道，就是在你还以为 *debugger* 是那种会发蓝光、用来电死倒霉虫子的设备的时候。你很可能会使用 `print` 语句，把程序的内部状态打印出来。C/C++ 程序员把这种做法称为 `printf` 调试（*printf debugging*，得名于 C 标准库函数 `printf()`）。

事实证明，`printf` 调试至今仍然是一种完全有效的做法——即便你已经知道调试器并不是夜里用来烤虫子的设备。尤其是在实时编程中，某些 bug 很难通过断点和监视窗口来追踪。有些 bug 与时序相关：它们只会在程序全速运行时才出现。还有些 bug 是由一连串复杂事件造成的，这些事件太长、太繁复，无法手动逐个追踪。在这些情况下，最强大的调试工具往往就是一系列打印语句。

每个游戏平台都有某种控制台或电传打字机（TTY）输出设备。下面是一些例子：

- 在 Linux 或 Win32 下运行的 C/C++ 控制台应用程序中，可以通过 `printf()`、`fprintf()` 或 C++ 标准库的 `iostream` 接口，将输出打印到 `stdout` 或 `stderr`，从而在控制台中产生输出。
- 遗憾的是，如果你的游戏在 Win32 下被构建为窗口应用程序，`printf()` 和 `iostream` 就不起作用，因为没有可用于显示输出的控制台。不过，如果程序运行在 Visual Studio 调试器下，调试器会提供一个调试控制台，可以通过 Win32 函数 `OutputDebugString()` 向其中打印信息。
- 在 PlayStation 平台上，有一个名为 Target Manager 的应用程序运行在 PC 上，并允许你在主机上启动程序。Target Manager 包含一组 TTY 输出窗口，游戏引擎可以向这些窗口打印消息。

因此，为调试目的打印信息几乎总是像在代码中到处添加 `printf()` 调用一样简单。不过，大多数游戏引擎会比这更进一步。在接下来的小节中，我们将考察大多数游戏引擎提供的各类打印设施。

### 10.1.1 使用 `OutputDebugString()` 进行格式化输出

Windows SDK 函数 `OutputDebugString()` 非常适合把调试信息打印到 Visual Studio 的 Debug Output 窗口中。然而，与 `printf()` 不同，`OutputDebugString()` 不支持格式化输出——它只能以字符数组的形式打印原始字符串。因此，大多数 Windows 游戏引擎都会用一个自定义函数对它进行封装，如下所示：

```cpp
#include <stdio.h>     // for va_list et al

#ifndef WIN32_LEAN_AND_MEAN
#define WIN32_LEAN_AND_MEAN 1
#endif
#include <windows.h>   // for OutputDebugString()

int VDebugPrintF(const char* format, va_list argList)
{
    const U32 MAX_CHARS = 1024;
    static char s_buffer[MAX_CHARS];

    int charsWritten
        = vsnprintf(s_buffer, MAX_CHARS, format, argList);

    // Now that we have a formatted string, call the
    // Win32 API.
    OutputDebugString(s_buffer);

    return charsWritten;
}

int DebugPrintF(const char* format, ...)
{
    va_list argList;
    va_start(argList, format);

    int charsWritten = VDebugPrintF(format, argList);

    va_end(argList);
    return charsWritten;
}
```

注意这里实现了两个函数：`DebugPrintF()` 接收可变长度参数列表（通过省略号 `...` 指定），而 `VDebugPrintF()` 接收一个 `va_list` 参数。这样做是为了让程序员能够基于 `VDebugPrintF()` 构建额外的打印函数。（不可能把省略号从一个函数传给另一个函数，但可以传递 `va_list`。）

### 10.1.2 详细程度

一旦你费力地在代码中精心选择的位置添加了一堆打印语句，最好能把它们保留下来，以便以后需要时再次使用。为了做到这一点，大多数引擎都会提供某种机制，用于通过命令行或在运行时动态控制详细程度（*verbosity*）级别。当详细程度级别处于最小值（通常是 0）时，只打印严重错误消息。当详细程度更高时，嵌入在代码中的更多打印语句就会开始参与输出。

实现这一机制的最简单方式，是把当前详细程度级别存储在一个全局整型变量中，比如叫作 `g_verbosity`。然后提供一个 `VerboseDebugPrintF()` 函数，它的第一个参数表示消息被打印所需的最低详细程度级别。这个函数可以实现如下：

```cpp
int g_verbosity = 0;

void VerboseDebugPrintF(int verbosity,
                        const char* format, ...)
{
    // Only print when the global verbosity level is
    // high enough.
    if (g_verbosity >= verbosity)
    {
        va_list argList;
        va_start(argList, format);

        VDebugPrintF(format, argList);

        va_end(argList);
    }
}
```

### 10.1.3 通道

将调试输出划分到不同的通道（*channels*）中也非常有用。例如，一个通道可能包含来自动画系统的消息，而另一个通道可能用于打印来自物理系统的消息。

在某些平台上，例如 PlayStation 的 Target Manager TTY，调试输出可以被定向到多个不同通道之一，每个通道显示在自己的标签页中。此外还有一个标签页会显示来自其他通道的所有输出的汇总。这样一来，开发者就很容易只关注自己想看的消息。在处理动画问题时，只需要切换到动画输出被定向到的对应通道，并忽略所有其他输出即可。当处理来源未知的一般性问题时，则可以查看 “all” TTY 通道来寻找线索。

Windows 等其他平台只提供单一的调试输出控制台。不过，即便在这些系统上，把输出划分为通道仍然很有帮助。每个通道的输出可以被分配不同颜色。也可以实现过滤器（*filters*），在运行时开启或关闭它们，并将输出限制到指定通道或通道集合。在这种模型下，如果开发者正在调试一个与动画相关的问题，就可以简单地过滤掉除动画通道之外的所有通道。

基于通道的调试输出系统可以很容易地实现：只需给调试打印函数添加一个额外的通道参数即可。通道可以编号；更好的做法是，通过 C/C++ 的 `enum` 声明为通道分配符号值。通道也可以使用字符串或经过哈希的字符串 ID 来命名。打印函数只需查询当前启用的通道列表，并且只有当指定通道位于其中时才打印消息。

如果你的通道数量不超过 32 或 64 个，那么使用 32 位或 64 位掩码来标识通道会很有帮助。这样，实现通道过滤器就像指定一个整数一样简单。当掩码中的某一位为 1 时，对应通道处于激活状态；当该位为 0 时，该通道被静音。

#### 10.1.3.1 使用 Redis 管理 TTY 通道

Naughty Dog 的开发者使用一个名为 Connector 的基于 Web 的界面，作为观察游戏引擎运行时发出的各种调试信息流的窗口。游戏会把调试文本输出到多个具名通道中，每个通道都与不同的引擎系统相关联（动画、渲染、AI、声音等）。这些数据流由一个轻量级 Redis 键值存储收集（关于 Redis 的更多信息，见 [226]）。Connector 界面允许用户通过任何 Web 浏览器轻松查看和过滤这些 Redis 数据。

### 10.1.4 将输出镜像到文件

把所有调试输出镜像到一个或多个日志文件中是个好主意（例如，每个通道一个文件）。这样可以在事后诊断问题。理想情况下，日志文件应该包含**所有**调试输出，并且不受当前详细程度级别和活动通道掩码的影响。这样，意外问题就可以通过检查最新的日志文件来捕获和追踪。

你可能还需要考虑在每次调用调试输出函数之后刷新日志文件，以确保如果游戏崩溃，日志文件不会丢失最后一缓冲区的输出。最后打印的数据通常对判断崩溃原因最有用，所以我们希望确保日志文件始终包含最新的输出。当然，刷新输出缓冲区可能代价较高。因此，只有在以下两种情况下才应该在每次调试输出调用后刷新缓冲区：（a）你没有进行大量日志记录；或（b）你发现这在特定平台上确实是必要的。如果认为刷新是必要的，也可以提供一个引擎配置选项，用于开启或关闭这一行为。

### 10.1.5 崩溃报告

有些游戏引擎会在游戏崩溃时生成特殊的文本输出和 / 或日志文件。在大多数操作系统中，可以安装一个顶层异常处理器，用来捕获大多数崩溃。在这个函数中，你可以打印各种有用的信息。你甚至可以考虑把崩溃报告通过电子邮件发送给整个程序团队。对于程序员来说，这可能非常有启发性：当他们看到美术和设计团队崩溃得如此频繁时，可能会重新产生一种调试任务的紧迫感！

下面是一些可以包含在崩溃报告中的信息示例：

- 崩溃发生时正在游玩的当前关卡。
- 崩溃发生时玩家角色的世界空间位置。
- 游戏崩溃时玩家的动画 / 动作状态。
- 崩溃发生时正在运行的玩法脚本。（如果脚本正是崩溃原因，这会特别有帮助！）
- 调用栈。大多数操作系统都提供了遍历调用栈的机制（尽管这些机制并非标准化，并且高度依赖平台）。借助这种设施，你可以打印出崩溃发生时调用栈上所有非内联函数的符号名称。
- 引擎中所有内存分配器的状态（空闲内存数量、碎片化程度等）。当 bug 是由低内存情况导致时，这类数据会很有帮助。
- 任何你认为可能与追踪崩溃原因有关的其他信息。
- 游戏崩溃瞬间的截图。

PlayStation 生态系统包含一个强大的崩溃报告系统。PS5 上的系统软件会检测崩溃，并将大量有关崩溃的信息上传到由 Sony Interactive Entertainment（SIE）托管和管理的 Recap 服务器。游戏开发者可以通过 RESTful API 访问 Recap 服务器，以获取崩溃统计信息、下载核心转储、TTY 输出、崩溃前发生情况的截图和视频，等等。
