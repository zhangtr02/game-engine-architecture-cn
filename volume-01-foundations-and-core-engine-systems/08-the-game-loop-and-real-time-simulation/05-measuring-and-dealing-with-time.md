# 8.5 测量和处理时间

在本节中，我们将研究不同类型的时间线和时钟之间一些微妙以及不那么微妙的区别，并看看它们在真实游戏引擎中是如何实现的。

## 8.5.1 帧率与时间增量

实时游戏的**帧率**（frame rate）描述的是一系列静态 3D 帧呈现给观看者的速度有多快。**赫兹**（Hertz, Hz）这个单位定义为每秒的周期数，可以用来描述任何周期性过程的频率。在游戏和电影中，帧率通常以**每秒帧数**（frames per second, FPS）来衡量；从各种实际用途来看，它与赫兹是一回事。电影传统上以 24 FPS 运行。北美和日本的游戏通常以 30 或 60 FPS 渲染，因为这是这些地区使用的 NTSC 彩色电视标准的自然刷新率。在欧洲以及世界其他大多数地区，游戏以 50 FPS 更新，因为这是 PAL 或 SECAM 彩色电视信号的自然刷新率。

两帧之间经过的时间量称为**帧时间**（frame time）、**时间增量**（time delta）或 **delta time**。最后一个术语很常见，因为两帧之间的持续时间通常在数学上表示为符号 `Δt`。（严格来说，`Δt` 实际上应该称为**帧周期**（frame period），因为它是**帧频率**（frame frequency）的倒数：`T = 1/f`。不过，游戏程序员在这种语境中几乎从不使用“周期”这个术语。）如果游戏正好以 30 FPS 渲染，那么它的 delta time 就是 1/30 秒，也就是 33.3 ms（毫秒）。在 60 FPS 下，delta time 是其一半，即 1/60 秒，也就是 16.6 ms。要真正知道游戏循环的一次迭代中经过了多少时间，我们需要对其进行测量。下面将看到具体做法。

这里还应该注意，毫秒是游戏中常见的时间测量单位。例如，我们可能会说动画系统运行需要 4 ms，这意味着它大约占用了整帧时间的 12%（`4 / 33.3 ≈ 0.12`）。其他常见单位包括秒和机器周期。稍后我们会更深入地讨论时间单位和时钟变量。

## 8.5.2 从帧率到速度

假设我们想让一艘飞船以每秒 40 米的恒定速度穿过游戏世界（或者在 2D 游戏中，可以指定为每秒 40 像素）。实现这一点的一种简单方式，是将飞船的速度 `v`（以米每秒为单位）乘以一帧的持续时间 `Δt`（以秒为单位），得到位置变化量 `Δx = vΔt`（以米每帧为单位）。然后可以将这个位置增量加到飞船当前的位置 `x1` 上，以得到下一帧的位置：`x2 = x1 + Δx = x1 + vΔt`。

这实际上是一种称为**显式欧拉法**（explicit Euler method）的简单**数值积分**（numerical integration）形式（见 Section 14.4.4）。只要物体速度大致保持恒定，它就能很好地工作。为了处理变化的速度，我们需要求助于稍微复杂一些的积分方法。不过，所有数值积分技术都会以某种方式使用经过的帧时间 `Δt`。因此可以说，游戏中物体的**感知速度**（perceived speeds）取决于帧持续时间 `Δt`。因此，游戏编程中的一个核心问题是确定一个合适的 `Δt` 值。在接下来的小节中，我们将讨论实现这一目标的各种方式。

### 8.5.2.1 老式 CPU 相关游戏

在许多早期电子游戏中，并没有尝试测量游戏循环期间经过了多少真实时间。程序员基本上会完全忽略 `Δt`，转而直接以每帧多少米（或像素，或其他距离单位）来指定物体速度。换句话说，也许是在无意中，他们指定的是 `Δx = vΔt` 形式的物体速度，而不是 `v` 本身。

这种简单做法的实际效果是，这些游戏中物体的感知速度完全取决于游戏在某个特定硬件上实际达到的帧率。如果这类游戏运行在一台 CPU 比原本编写目标机器更快的计算机上，游戏看起来就会像在快进一样运行。因此，我将这类游戏称为 **CPU 相关游戏**（CPU-dependent games）。

一些较早的 PC 提供了一个 “Turbo” 按钮，用于支持这类游戏。当按下 Turbo 按钮时，PC 会以最快速度运行，但 CPU 相关游戏也会快进运行。当没有按下 Turbo 按钮时，PC 会模拟较早一代 PC 的处理器速度，使为那些 PC 编写的 CPU 相关游戏能够正常运行。

### 8.5.2.2 基于经过时间进行更新

为了让游戏不依赖 CPU，我们必须以某种方式测量 `Δt`，而不是简单地忽略它。这件事相当直接。我们只需要读取 CPU 高精度计时器的值两次：一次在帧开始时，另一次在帧结束时。然后相减，就能得到刚刚过去这一帧的 `Δt` 的精确测量值。随后，这个 delta 会被提供给所有需要它的引擎子系统；可以将它传递给游戏循环内部调用的每个函数，也可以将其存储在全局变量中，或封装在某种单例类中。（我们会在 Section 8.5.3 中更详细地描述 CPU 高精度计时器。）

上述方法被许多游戏引擎使用。事实上，我甚至想大胆地说，大多数游戏引擎都使用这种方法。然而，这种技术存在一个大问题：我们使用第 `k` 帧期间测得的 `Δt` 值，作为即将到来的第 `k + 1` 帧持续时间的估计。这未必非常准确。（正如投资领域常说的：“过去的表现并不保证未来的结果。”）下一帧可能发生某些事情，使其耗时远多于（或远少于）当前帧。我们将这类事件称为**帧率尖峰**（frame-rate spike）。

使用上一帧的 delta 作为下一帧的估计值，可能会产生一些非常真实的负面影响。例如，如果不小心，它可能让游戏陷入糟糕帧时间的“恶性循环”。假设我们的物理模拟在每 33.3 ms 更新一次（即 30 Hz）时最稳定。如果出现一帧糟糕帧，耗时 57 ms，那么在下一帧中，我们可能会错误地两次步进物理系统，大概是为了“补上”已经过去的 57 ms。这两次步进大约需要普通步进两倍的时间，从而导致下一帧至少与这一帧一样糟，甚至可能更糟。这只会加剧并延长问题。

### 8.5.2.3 使用滑动平均

游戏循环往往至少具有一定程度的逐帧连贯性，这是事实。如果某一帧中摄像机指向一条走廊，而走廊里有大量绘制开销很高的物体，那么下一帧它很可能仍然指向那条走廊。因此，一种合理方法是对少量帧的帧时间测量值取平均，并将其作为下一帧 `Δt` 的估计值。这样可以让游戏适应变化的帧率，同时缓和瞬时性能尖峰带来的影响。平均区间越长，游戏对变化帧率的响应越不灵敏，但尖峰的影响也会越小。

### 8.5.2.4 控制帧率

我们可以通过反过来思考这个问题，完全避免使用上一帧 `Δt` 作为当前帧持续时间估计所带来的不准确性。与其猜测下一帧持续时间是多少，不如尝试**保证**每一帧持续时间都正好为 33.3 ms（如果以 60 FPS 运行，则为 16.6 ms）。为此，我们像之前一样测量当前帧的持续时间。如果测得的持续时间小于理想帧时间，就让主线程休眠，直到目标帧时间已经过去。如果测得的持续时间超过理想帧时间，那么我们就只能“认栽”，再等待一个完整帧时间过去。这称为**帧率控制**（frame-rate governing）。

显然，只有当游戏的平均帧率合理接近目标帧率时，这种方法才有效。如果游戏由于频繁出现“慢”帧而在 30 FPS 和 15 FPS 之间来回跳变，那么游戏质量会显著下降。因此，仍然最好将所有引擎系统设计为能够处理任意帧持续时间。在开发期间，可以让引擎保持在“可变帧率”模式下，一切都会按预期工作。之后，当游戏越来越接近稳定达到目标帧率时，就可以开启帧率控制，并开始获得它的好处。

保持帧率一致可能很重要，原因有很多。一些引擎系统，例如物理模拟中使用的数值积分器，在以恒定频率更新时工作得最好。一致的帧率看起来也更好，并且正如下一节将看到的，它还可以用于避免当视频缓冲区以与显示器刷新率不匹配的频率更新时可能出现的**画面撕裂**（tearing，见 Section 8.5.2.5）。

此外，当经过的帧时间保持一致时，像**记录与回放**（record and playback）这样的功能会变得更加可靠。顾名思义，记录与回放功能允许玩家的游戏过程被记录下来，并在稍后以完全相同的方式播放。这可以是一个有趣的游戏功能，也是一种有价值的测试和调试工具。例如，难以发现的 bug 可以通过简单回放一段能展示该 bug 的已记录游戏过程来复现。

为了实现记录与回放，我们会记录游戏过程中发生的每一个相关事件，并将每个事件连同准确时间戳保存到列表中。随后，可以使用完全相同的时序、相同的初始条件，以及相同的初始随机种子来回放这份事件列表。理论上，这应该产生与原始游玩过程无法区分的游戏体验。然而，如果帧率不一致，事情可能不会以完全相同的顺序发生。这会导致“漂移”（drift），很快你的 AI 角色就会在本应后撤时进行包抄。

### 8.5.2.5 屏幕撕裂与垂直同步

一种称为**屏幕撕裂**（screen tearing）的视觉异常，会在屏幕只被视频硬件部分“绘制”完成时，就将后缓冲区与前缓冲区交换时发生。当撕裂发生时，屏幕的一部分显示新图像，而剩余部分仍显示旧图像。为了避免撕裂，许多渲染引擎会等到显示器的**垂直消隐间隔**（vertical blanking interval）到来后再交换缓冲区。

较早的 CRT 显示器和电视会通过一束电子束从左到右、从上到下扫描，并激发屏幕上的荧光粉，从而“绘制”内存帧缓冲区中的内容。在这类显示器上，垂直消隐间隔是电子枪被“消隐”（关闭）的时间段，此时它会被重置到屏幕左上角。现代 LCD、等离子和 LED 显示器不再使用电子束，也不需要在完成一帧最后一条扫描线绘制和下一帧第一条扫描线绘制之间留出任何时间。不过，垂直消隐间隔仍然存在，部分原因是视频标准是在 CRT 占主流时建立的，部分原因则是为了支持较旧显示设备。

等待垂直消隐间隔称为**垂直同步**（v-sync）。它实际上只是另一种帧率控制形式，因为它有效地将主游戏循环的帧率钳制为屏幕刷新率的整数倍。例如，在刷新率为 60 Hz 的 NTSC 显示器上，游戏的真实更新率实际上会被量化为 1/60 秒的整数倍。如果帧与帧之间经过的时间超过 1/60 秒，那么必须等到下一个垂直消隐间隔，也就是等待 2/60 秒（30 FPS）。如果错过两个垂直消隐间隔，那么必须总共等待 3/60 秒（20 FPS），以此类推。此外，即使游戏同步到垂直消隐间隔，也不要对游戏帧率作出过多假设；如果游戏支持 PAL 和 SECAM 标准，就必须记住这些标准基于 50 Hz 更新率，而不是 60 Hz。

一些现代显示设备支持一种称为**可变刷新率**（variable refresh rate, VRR）的技术。该技术允许显示设备的刷新率与游戏渲染帧的速率同步，从而消除撕裂并提升渲染效率。

## 8.5.3 使用高精度计时器测量真实时间

我们已经大量讨论了如何测量每一帧中经过的真实“挂钟”时间。本节将详细研究这类时间测量是如何完成的。

大多数操作系统都会提供查询系统时间的函数，例如 C 标准库函数 `time()`。然而，这类函数并不适合测量实时游戏中的经过时间，因为它们无法提供足够的分辨率。例如，`time()` 返回一个整数，表示自 1970 年 1 月 1 日午夜以来经过的秒数，因此它的分辨率是一秒；考虑到一帧只需要几十毫秒执行，这个粒度远远太粗。

所有现代 CPU 都包含一个**高精度计时器**（high-resolution timer），通常实现为一个硬件寄存器，用于统计自处理器上一次通电或重置以来经过的 CPU 周期数（或其某种倍数）。在游戏中测量经过时间时，应该使用这个计时器，因为它的分辨率通常只有若干 CPU 周期的量级。例如，在 3 GHz Pentium 处理器上，高精度计时器每个 CPU 周期递增一次，也就是每秒超过 30 亿次。因此，高精度计时器的分辨率是 `1 / 3 billion = 3.33 × 10^-10` 秒，也就是 0.333 ns（三分之一纳秒）。这对游戏中的所有时间测量需求来说都绰绰有余。

不同微处理器和不同操作系统提供了不同方式来查询高精度计时器。在 Pentium 上，可以使用一个名为 `rdtsc`（read time-stamp counter）的特殊指令；不过 Win32 API 将这个功能包装成一对函数：`QueryPerformanceCounter()` 读取 64 位计数器寄存器，而 `QueryPerformanceFrequency()` 返回当前 CPU 每秒的计数器递增次数。在 PowerPC 架构上，例如 Xbox 360 和 PlayStation 3 中使用的芯片，可以使用 `mftb`（move from time base register）指令读取两个 32 位时间基准寄存器；而在其他 PowerPC 架构上，则改用 `mfspr`（move from special-purpose register）指令。

为了确保不会太频繁地回绕，大多数处理器上的 CPU 高精度计时器寄存器宽度为 64 位。64 位无符号整数的最大可能值为 `0xFFFFFFFFFFFFFFFF ≈ 1.8 × 10^19` 个时钟 tick。因此，在 3 GHz Pentium 处理器上，如果每个 CPU 周期都更新一次高精度计时器，那么该寄存器大约每 195 年才会回绕到零——这显然不是我们需要过分担心的情况。相比之下，在 3 GHz 下，32 位整数时钟只需大约 1.4 秒就会回绕。

### 8.5.3.1 高精度时钟漂移

需要注意，即使通过高精度计时器进行的时间测量，在某些情况下也可能不准确。例如，在某些多核处理器上，每个核心上的高精度计时器彼此独立，它们可能会（而且确实会）逐渐漂移。如果你试图比较不同核心上获取的绝对计时器读数，可能会得到一些奇怪结果，甚至出现负的时间增量。一定要留意这类问题。

## 8.5.4 时间单位与时钟变量

每当我们在游戏中测量或指定时间持续时间时，都需要做出两个选择：

1. 应该使用什么**时间单位**？我们是希望以秒、毫秒、机器周期，还是其他某种单位来存储时间？

2. 应该使用什么**数据类型**来存储时间测量值？应该采用 64 位整数、32 位整数，还是 32 位浮点变量？

这些问题的答案取决于给定测量值的预期用途。这又引出了另外两个问题：我们需要多少精度？以及我们期望能够表示多大的量级范围？

### 8.5.4.1 64 位整数时钟

我们已经看到，以机器周期为单位测量的 64 位无符号整数时钟，既支持极高精度（在 3 GHz CPU 上，单个周期持续 0.333 ns），又支持很宽的量级范围（在 3 GHz 下，64 位时钟大约每 195 年才回绕一次）。因此，假设你负担得起 64 位存储空间，这是最灵活的时间表示方式。

### 8.5.4.2 32 位整数时钟

当需要以高精度测量相对较短的持续时间时，可以使用以机器周期为单位的 32 位整数时钟。例如，为了分析某段代码块的性能，我们可以这样做：

    // 获取时间快照。
    U64 begin_ticks = readHiResTimer();

    // 这是我们希望测量性能的代码块。
    doSomething();
    doSomethingElse();
    nowReallyDoSomething();

    // 测量持续时间。
    U64 end_ticks = readHiResTimer();
    U32 dt_ticks = static_cast<U32>(end_ticks - begin_ticks);

    // 现在使用或缓存 dt_ticks 的值……

注意，我们仍然将原始时间测量值存储在 64 位整数变量中。只有时间增量 `dt` 会被存储在 32 位变量中。这可以绕开 32 位边界处潜在的回绕问题。例如，如果 `begin_ticks = 0x12345678FFFFFFB7`，而 `end_ticks = 0x1234567900000039`，那么如果我们在相减之前先把各个时间测量值截断为 32 位，就会测得一个负的时间增量。

### 8.5.4.3 32 位浮点时钟

另一种常见方法，是以浮点格式存储相对较小的时间增量，单位为秒。为此，只需将以 CPU 周期为单位测得的持续时间除以 CPU 时钟频率（单位为每秒周期数）。例如：

    // 一开始假设理想帧时间（30 FPS）。
    F32 dt_seconds = 1.0f / 30.0f;

    // 通过读取当前时间来预热循环。
    U64 begin_ticks = readHiResTimer();

    while (true) // 主游戏循环
    {
        runOneIterationOfGameLoop(dt_seconds);

        // 再次读取当前时间，并计算 delta。
        U64 end_ticks = readHiResTimer();

        // 检查单位：秒 = tick / (tick/秒)
        dt_seconds = (F32)(end_ticks - begin_ticks)
                   / (F32)getHiResTimerFrequency();

        // 将 end_ticks 作为下一帧的新 begin_ticks。
        begin_ticks = end_ticks;
    }

再次注意，我们必须在将两个 64 位时间测量值转换为浮点格式之前先对它们相减。这可以确保不会把过大的量级存入 32 位浮点变量。

### 8.5.4.4 浮点时钟的局限

回想一下，在 32 位 IEEE 浮点数中，尾数的 23 位会通过指数在数值的整数部分和小数部分之间动态分配（见 Section 3.3.1.4）。较小的量级只需要少量位，因此会为小数部分留下大量精度位。但一旦时钟的量级变得过大，整数部分就会吞掉更多位，留给小数部分的位数就会减少。最终，甚至整数部分的最低有效位也会变成隐式零。这意味着，在浮点时钟变量中存储**长持续时间**时必须小心。如果我们持续追踪自游戏启动以来经过了多少时间，浮点时钟最终会变得不准确，甚至达到无法使用的程度。

浮点时钟通常只用于存储相对较短的时间增量，最多测量几分钟，更常见的是只测量一帧或更短时间。如果在游戏中使用绝对值形式的浮点时钟，就需要定期将时钟重置为零，以避免大量级不断累积。

### 8.5.4.5 其他时间单位

有些游戏引擎会以游戏自定义单位来指定计时值。这种单位要足够细粒度，以允许使用整数格式（而不是必须使用浮点格式）；要足够精确，以便对引擎内部的广泛应用有用；同时也要足够大，使 32 位时钟不会过于频繁地回绕。一种常见选择是使用 `1/300` 秒作为时间单位。这种方式效果很好，因为：（a）它对于许多用途来说足够细粒度；（b）它每 165.7 天才回绕一次；（c）它同时是 NTSC 和 PAL 刷新率的整数倍。60 FPS 的一帧持续 5 个这样的单位，而 50 FPS 的一帧持续 6 个单位。

另一种可能性是以整数**毫秒**为单位存储时间。这比 `1/300` 秒时间单位提供了好得多的分辨率，同时仍然允许在有限位数中表示合理范围的时间：一个以毫秒为单位存储时间的无符号 32 位整数会在 49.7 天后回绕——对于大多数用途来说，这已经有足够的范围。

## 8.5.5 处理断点

当游戏命中断点时，它的循环会停止运行，调试器接管控制。然而，如果游戏运行在与调试器相同的计算机上，那么 CPU 会继续运行，真实时间时钟也会继续累积周期。当你在断点处检查代码时，大量挂钟时间可能会过去。当你允许程序继续运行时，这可能会导致测得的帧时间达到许多秒，甚至数分钟或数小时！

显然，如果我们允许如此巨大的 delta time 被传递给引擎中的子系统，就会发生糟糕事情。如果运气好，游戏可能会在单帧中猛然向前跳跃数秒后继续正常运行。更糟的是，游戏可能直接崩溃。

可以使用一种简单方法绕过这个问题。在主游戏循环中，如果我们测得的帧时间超过某个预定义上限（例如 1 秒），就可以假定我们刚刚从断点恢复执行，并将 delta time 人为设置为 1/30 或 1/60 秒（或者目标帧率对应的任意值）。实际上，为了避免测得帧持续时间出现巨大尖峰，游戏会在这一帧中被锁定到目标帧率。

    // 一开始假设理想 dt（30 FPS）。
    F32 dt = 1.0f / 30.0f;

    // 通过读取当前时间来预热循环。
    U64 begin_ticks = readHiResTimer();

    while (true) // 主游戏循环
    {
        updateSubsystemA(dt);
        updateSubsystemB(dt);
        // ...
        renderScene();
        swapBuffers();

        // 再次读取当前时间，并计算
        // 下一帧 delta time 的估计值。
        U64 end_ticks = readHiResTimer();

        dt = (F32)(end_ticks - begin_ticks)
           / (F32)getHiResTimerFrequency();

        // 如果 dt 过大，说明我们一定是从断点恢复执行了——
        // 这一帧锁定到目标帧率。
        if (dt > 1.0f)
        {
            dt = 1.0f / 30.0f;
        }

        // 将 end_ticks 作为下一帧的新 begin_ticks。
        begin_ticks = end_ticks;
    }
