### 3.2 捕获与处理错误

在游戏引擎中，捕获和处理错误条件有许多不同方式。作为游戏程序员，理解这些不同机制、它们各自的优缺点，以及应该在什么时候使用哪一种机制，是非常重要的。

#### 3.2.1 错误类型

在任何软件项目中，错误条件基本上可以分为两类：**用户错误**和**程序员错误**。**用户错误**发生在程序的用户做了不正确的事情时，例如输入了无效内容、试图打开一个不存在的文件，等等。**程序员错误**则是代码本身存在 bug 所导致的结果。虽然它可能由用户的某个操作触发，但程序员错误的本质在于：如果程序员没有犯错，这个问题本可以避免；而且用户有合理期待，认为程序**本应该**优雅地处理这种情况。

当然，“用户”的定义会根据上下文而变化。在游戏项目的语境中，用户错误大致可以分成两类：由玩游戏的人造成的错误，以及由开发过程中制作游戏的人造成的错误。重要的是，要弄清楚某个特定错误影响的是哪一类用户，并以适当方式处理该错误。

实际上，还有第三类用户——你团队中的其他程序员。（如果你正在编写一款游戏中间件软件，比如 Havok 或 OpenGL，那么这第三类用户还会扩展到全世界使用你库的其他程序员。）在这里，用户错误和程序员错误之间的界限会变得模糊。假设程序员 A 写了一个函数 `f()`，程序员 B 尝试调用它。如果 B 使用无效参数调用 `f()`（例如空指针，或越界数组索引），那么从程序员 A 的角度看，这可以被视为用户错误；但从 B 的角度看，它又是程序员错误。（当然，也可以认为程序员 A 本应该预料到可能会传入无效参数，并且应该优雅地处理它们，所以这个问题实际上是 A 的程序员错误。）这里需要记住的关键点是，用户和程序员之间的界限会随上下文而改变——它很少是非黑即白的区别。

#### 3.2.2 处理错误

在处理错误时，这两类错误的需求差异很大。最好尽可能优雅地处理**用户错误**：向用户显示一些有帮助的信息，然后允许他们继续工作——或者在游戏中，允许他们继续玩。另一方面，**程序员错误**不应该采用优雅的“通知并继续”策略来处理。相反，通常最好中止程序，并提供详细的底层调试信息，使程序员能够快速定位并修复问题。在理想情况下，所有程序员错误都应该在软件发布给公众之前被发现并修复。

##### 3.2.2.1 处理玩家错误

当“用户”是正在玩游戏的人时，错误显然应该在游戏玩法的上下文中处理。例如，如果玩家在没有弹药时试图重新装填武器，一个音频提示和/或动画就可以向玩家表明这个问题，而不必把他们“从游戏中拉出来”。

##### 3.2.2.2 处理开发者错误

当“用户”是正在制作游戏的人，例如美术、动画师或游戏设计师时，错误可能由某种无效资产导致。例如，一段动画可能关联到了错误的骨架，纹理尺寸可能不正确，或者音频文件可能使用了不受支持的采样率。对于这类**开发者错误**，存在两种相互竞争的观点。

一方面，防止坏的游戏资产长期存在似乎很重要。一个游戏通常包含成千上万的资产，而一个有问题的资产可能会“丢失”在其中；这样就有风险让这个坏资产一直存活到最终发布的游戏中。如果把这种观点推向极端，那么处理坏游戏资产的最佳方式，就是只要遇到哪怕一个有问题的资产，就阻止整个游戏运行。这当然会强烈促使创建无效资产的开发者立即删除或修复它。

另一方面，游戏开发是一个混乱且迭代的过程，第一次就生成“完美”的资产其实非常少见。按照这种思路，游戏引擎应该对几乎任何可以想象的问题都保持鲁棒性，使得即使面对完全无效的游戏资产数据，工作也能继续进行。但这也并不理想，因为游戏引擎会被大量错误捕获和错误处理代码膨胀，而这些代码在开发节奏稳定下来、游戏发布之后并不需要。同时，带着“坏”资产发布产品的概率也会变得过高。

根据我的经验，最佳做法是在这两个极端之间寻找中间道路。当开发者错误发生时，我倾向于**让错误变得显眼**，然后允许团队在问题存在的情况下继续工作。仅仅因为某个开发者试图向游戏中添加一个无效资产，就阻止团队中所有其他开发者工作，是极其昂贵的。游戏工作室支付给员工的薪酬并不低，当多个团队成员遭遇停工时，成本会随着被阻止工作的人数成倍增加。当然，只有在可行的情况下才应该以这种方式处理错误，不能为此花费过多工程时间，或让代码膨胀。

举个例子，假设某个特定网格无法加载。在我看来，最好的做法是在游戏世界中该网格原本应该出现的位置绘制一个醒目的红色大盒子，也许还可以在每个盒子上方悬浮一行文字，例如 “Mesh blah-dee-blah failed to load.” 这比把一条很容易被忽略的消息打印到错误日志中要好。而且它**远远好于**直接让游戏崩溃，因为一旦游戏崩溃，在那个网格引用被修复之前，就没有人能够继续工作。当然，对于特别严重的问题，直接输出错误信息并崩溃也是可以的。不存在适用于所有问题的银弹；你对某个具体情况应该采用哪种错误处理方式的判断，会随着经验增长而提升。

##### 3.2.2.3 处理程序员错误

检测和处理程序员错误（也就是 bug）的最佳方式，通常是在源代码中嵌入错误检查代码，并安排失败的错误检查中止程序。这种机制被称为**断言系统**；我们将在第 3.2.3.3 节中详细讨论断言。当然，正如前面所说，一个程序员的用户错误可能是另一个程序员的 bug；因此，断言并不总是处理每一种程序员错误的正确方式。在断言和更优雅的错误处理技术之间做出明智选择，是一种会随着时间逐渐培养出来的技能。

#### 3.2.3 错误检测与处理的实现

我们已经看过一些关于错误处理的哲学性方法。现在，让我们把注意力转向程序员在实现错误检测和错误处理代码时可以选择的方案。

##### 3.2.3.1 错误返回码

处理错误的一种常见方法，是从最初检测到问题的函数中返回某种失败码。这可以是一个表示成功或失败的布尔值，也可以是一个“不可能”的值，即位于正常返回结果范围之外的值。例如，一个返回正整数或正浮点值的函数，可以返回负值表示发生了错误。比布尔值或“不可能”返回值更好的做法，是将函数设计为返回一个枚举值，用来表示成功或失败。这样可以清晰地把错误码与函数输出结果区分开来，并且可以在失败时指明问题的确切性质，例如 `enum Error { kSuccess, kAssetNotFound, kInvalidRange, ... }`。

调用函数应该拦截错误返回码并采取适当行动。它可能会立即处理该错误。或者，它可能绕过这个问题，完成自己的执行，然后把错误码继续传递给调用它的函数。

##### 3.2.3.2 异常

错误返回码是一种简单而可靠的方式，用于传达和响应错误条件。然而，错误返回码也有缺点。错误返回码最大的问题也许在于：检测到错误的函数，可能与能够处理该问题的函数完全无关。在最坏情况下，调用栈深处第 40 层的某个函数可能检测到一个问题，而这个问题只能由顶层游戏循环或 `main()` 来处理。在这种情况下，调用栈上的 40 个函数都必须被编写成能够把合适的错误码一路向上传递回顶层错误处理函数。

解决这个问题的一种方式是抛出异常。**异常处理**是 C++ 的一个非常强大的特性。它允许检测到问题的函数把错误传达给代码的其他部分，而不需要知道哪个函数可能处理该错误。当异常被抛出时，与该错误相关的信息会被放入一个由程序员选择的数据对象中，这个对象称为**异常对象**。随后调用栈会自动展开，寻找一个调用函数，该函数把调用包裹在 `try-catch` 块中。如果找到了 `try-catch` 块，异常对象会与所有可能的 `catch` 子句进行匹配；如果找到匹配项，就执行对应 `catch` 的代码块。在栈展开过程中，任何自动变量的析构函数都会按需要被调用。

以如此干净的方式分离错误检测和错误处理，当然很有吸引力；对于某些软件项目来说，异常处理也是一个很好的选择。然而，异常处理确实会给程序增加一些开销。任何包含 `try-catch` 块的函数，其栈帧都必须被扩展，以包含栈展开过程所需的额外信息。此外，如果程序中的任何一个函数（或程序链接到的某个库中的任何一个函数）使用异常处理，那么整个程序都必须使用异常处理——编译器无法知道当你抛出异常时，调用栈上方可能有哪些函数。

话虽如此，我们可以把一个或多个使用异常处理的库“沙盒化”，以避免整个游戏引擎都必须在启用异常的情况下编写。要做到这一点，可以把对相关库的所有 API 调用都包装到一些函数中，而这些函数实现在一个启用了异常处理的翻译单元中。每个这样的函数都会在 `try/catch` 块中捕获所有可能的异常，并将它们转换为错误返回码。因此，任何链接到你的包装器库的代码，都可以安全地禁用异常处理。

可以说，比开销问题更重要的是这样一个事实：异常在某些方面并不比 `goto` 语句更好。微软和 Fog Creek Software 的 Joel Spolsky 认为，异常实际上比 `goto` 更糟，因为它们在源代码中并不容易看见。一个既不抛出也不捕获异常的函数，仍然可能参与栈展开过程，只要它位于调用栈中某些函数之间。而展开过程本身并不完美：除非程序员考虑到异常可能被抛出的每一种方式，并适当地处理它，否则软件很容易被留在无效状态。这会让编写健壮软件变得困难。当存在抛出异常的可能性时，代码库中几乎每一个函数都需要足够健壮，以应对地毯被从脚下抽走、所有局部对象都在函数调用中途被销毁的情况。

异常处理的另一个问题是它的成本。虽然理论上现代异常处理框架在没有错误发生的情况下不会引入额外运行时开销，但实践中并不一定如此。例如，当异常发生时，编译器为栈展开而添加到函数中的代码，往往会导致整体代码尺寸增加。这可能降低 I-cache 性能，或者导致编译器决定不内联一个它原本会内联的函数。

显然，有一些相当有力的理由支持在游戏引擎中完全关闭异常处理。这正是 Naughty Dog 采用的方法，也是我在 Electronic Arts 和 Midway 参与的大多数项目采用的方法。Mike Acton 在担任 Insomniac Games 引擎总监期间，也多次明确表示反对在运行时游戏代码中使用异常处理。JPL 和 NASA 也不允许在其任务关键型嵌入式软件中使用异常处理，大概也是出于我们在游戏行业中倾向于避免异常的相同原因。话虽如此，具体情况当然会有所不同。不存在完美的工具，也不存在做任何事情都绝对正确的方式。只要明智地使用，异常确实可以让代码更容易编写和使用；只是要小心！

网上有许多关于这一主题的有趣文章。这里有几个不错的讨论串，涵盖了这场争论双方的大多数关键问题：[116]、[117] 和 [118]。

**异常与 RAII。**

“资源获取即初始化”（resource acquisition is initialization，RAII）模式（见第 3.1.1.6 节）经常与**异常处理**结合使用：构造函数尝试获取所需资源，如果失败就抛出异常。这样做是为了避免在对象创建之后再用 `if` 检查对象状态——如果构造函数返回且没有抛出异常，我们就能确定资源已经成功获取。

不过，RAII 模式即使没有异常也可以使用。它只需要一点纪律：在每个新的资源对象第一次被创建时检查其状态。此后，RAII 的所有其他好处仍然可以获得。（在某些资源获取失败的情况下，也可以用断言失败来代替异常。）

##### 3.2.3.3 断言

**断言**是一行用于检查表达式的代码。如果表达式求值为真，什么也不会发生。但如果表达式求值为假，程序会停止，打印一条消息，并在可能的情况下调用调试器。

断言检查的是程序员的假设。它们像 bug 的地雷。它们在代码最初编写时检查代码，以确保代码正常工作。它们还确保原始假设在很长一段时间内持续成立，即使周围代码一直在变化和演化。例如，如果某个程序员修改了原本能工作的代码，却意外违反了其原始假设，他们就会踩到地雷。这会立即通知程序员问题所在，并允许他们以最小代价纠正情况。没有断言时，bug 往往会“藏起来”，并在之后以难以追踪、耗时很长的方式表现出来。而在代码中嵌入断言后，bug 会在被引入的那一刻暴露出来——这通常也是修复问题的最佳时机，因为导致问题的代码修改还新鲜地留在程序员脑中。Steve Maguire 在他那本必读书 *Writing Solid Code* [39] 中对断言进行了深入讨论。

断言检查的成本通常在开发期间是可以接受的，但在游戏发布前去除断言，必要时可以换回那一点点关键性能。因此，断言通常会被实现成能够在非调试构建配置中从可执行文件中剥离出去。在 C 中，标准库头文件 `<assert.h>` 提供了一个 `assert()` 宏；在 C++ 中，它由 `<cassert>` 头文件提供。

标准库对 `assert()` 的定义会使它在调试构建中被定义（即定义了 `DEBUG` 预处理器符号的构建），并在非调试构建中被剥离（即定义了 `NDEBUG` 预处理器符号的构建）。在游戏引擎中，你可能希望更细粒度地控制哪些构建配置保留断言，以及哪些构建配置剥离断言。例如，你的游戏可能不仅支持调试构建和开发构建配置，还可能有一个启用全局优化的发布构建，甚至可能还有一个供 profile-guided optimization 工具使用的 PGO 构建（见第 2.2.4 节）。或者，你可能还想定义不同“风味”的断言——有些断言即使在游戏的发布版本中也始终保留，而另一些则会从非发布构建中剥离。基于这些原因，让我们看看如何使用 C/C++ 预处理器实现自己的 `ASSERT()` 宏。

**断言实现。**

断言通常通过以下几部分组合实现：一个 `#defined` 宏，该宏求值为一个 `if/else` 子句；一个在断言失败时（表达式求值为 `false`）被调用的函数；以及一小段汇编代码，用于中止程序，并在附加调试器时进入调试器。下面是一个典型实现：

```cpp
#if ASSERTIONS_ENABLED

// define some inline assembly that causes a break
// into the debugger -- this will be different on each
// target CPU
#define debugBreak() asm { int 3 }

// check the expression and fail if it is false
#define ASSERT(expr) \
    if (expr) { } \
    else \
    { \
        reportAssertionFailure(#expr, \
            __FILE__, __LINE__); \
        debugBreak(); \
    }

#else

#define ASSERT(expr) // evaluates to nothing

#endif
```

让我们拆解这个定义，看看它是如何工作的：

- 外层的 `#if/#else/#endif` 用于从代码库中剥离断言。当 `ASSERTIONS_ENABLED` 非零时，`ASSERT()` 宏会以完整形式被定义，代码中的所有断言检查都会被包含进程序中。但当断言被关闭时，`ASSERT(expr)` 不会求值为任何东西，代码中它的所有实例都会被有效移除。
- `debugBreak()` 宏会求值为某种汇编语言指令，用来使程序中止，并让调试器接管（如果连接了调试器）。这在不同 CPU 上会有所不同，但通常是一条汇编指令。
- `ASSERT()` 宏本身使用完整的 `if/else` 语句定义，而不是一个孤立的 `if`。这样做是为了让该宏可以在任何上下文中使用，甚至可以用于其他没有花括号包裹的 `if/else` 语句内部。

下面展示如果 `ASSERT()` 使用孤立的 `if` 来定义，会发生什么：

```cpp
// WARNING: NOT A GOOD IDEA!
#define ASSERT(expr) if (!(expr)) debugBreak()

void f()
{
    if (a < 5)
        ASSERT(a >= 0);
    else
        doSomething(a);
}
```

它会展开成下面这个**错误**代码：

```cpp
void f()
{
    if (a < 5)
        if (!(a >= 0))
            debugBreak();
        else // oops! bound to the wrong if()!
            doSomething(a);
}
```

- `ASSERT()` 宏中的 `else` 子句做了两件事。它会向程序员显示某种消息，说明哪里出了问题，然后进入调试器。请注意，消息显示函数的第一个参数使用了 `#expr`。井号（`#`）预处理器运算符会把表达式 `expr` 转换成字符串，从而允许它作为断言失败消息的一部分被打印出来。
- 还要注意 `__FILE__` 和 `__LINE__` 的使用。这些编译器定义的宏会神奇地包含它们所在 `.cpp` 文件的名称，以及它们所在代码行的行号。把它们传给我们的消息显示函数后，就可以打印出问题发生的确切位置。

我强烈建议在代码中使用断言。不过，了解它们的性能成本也很重要。你可能希望考虑定义两类断言宏。普通的 `ASSERT()` 宏可以在**所有**构建中保持启用，这样即使不在调试模式下运行，也能轻松捕获错误。第二个断言宏，也许可以称为 `SLOW_ASSERT()`，则只在调试构建中启用。这个宏可以用于那些断言检查成本过高、不适合包含在发布构建中的地方。显然，`SLOW_ASSERT()` 的用途较低，因为它会从测试人员每天游玩的游戏版本中剥离出去。但至少当程序员调试代码时，这些断言会变为启用状态。

正确使用断言也极其重要。它们应该用于捕获程序本身中的 bug——**绝不**应该用于捕获用户错误。此外，断言失败时应该始终导致整个游戏中止。允许测试人员、美术、设计师和其他非工程人员跳过断言通常是个坏主意。（这有点像“狼来了”的故事：如果断言可以被跳过，那么它们最终就失去所有意义。）换句话说，断言应该只用于捕获致命错误。如果可以在断言之后继续运行，那么更好的做法可能是用其他方式通知用户该错误，例如屏幕上的消息，或者某些难看的亮橙色 3D 图形。

**编译期断言。**

到目前为止，我们讨论的断言有一个弱点：它们内部编码的条件只会在运行时被检查。我们必须运行程序，而且相关代码路径必须真正**执行**，断言条件才会被检查。

有时，我们在断言中检查的条件涉及完全在编译期就已知的信息。例如，假设我们正在定义一个结构体，由于某种原因它必须正好是 128 字节大小。我们希望添加一个断言，使得如果另一个程序员（或未来版本的你自己）修改了这个结构体的大小，编译器就会给出错误消息。换句话说，我们希望写出类似下面这样的代码：

```cpp
struct NeedsToBe128Bytes
{
    U32     m_a;
    F32     m_b;
    // etc.
};

// sadly this doesn't work...
ASSERT(sizeof(NeedsToBe128Bytes) == 128);
```

当然，问题在于 `ASSERT()`（或 `assert()`）宏必须在运行时可执行，而你甚至不能在函数定义之外的 `.cpp` 文件全局作用域中放置可执行代码。这个问题的解决方案是**编译期断言**，也称为**静态断言**。

从 C++11 开始，标准库为我们定义了一个名为 `static_assert()` 的宏。因此，我们可以把上面的例子重写如下：

```cpp
struct NeedsToBe128Bytes
{
    U32     m_a;
    F32     m_b;
    // etc.
};

static_assert(sizeof(NeedsToBe128Bytes) == 128,
    "wrong size");
```

如果你使用的不是 C++11 或更新版本，也可以自己实现 `STATIC_ASSERT()` 宏。它可以通过许多不同方式实现，但基本思想始终相同：这个宏会在你的代码中放置一个声明，该声明（a）在文件作用域是合法的，（b）在编译期而不是运行时对目标表达式求值，并且（c）如果表达式为假，则产生编译器错误。某些定义 `STATIC_ASSERT()` 的方法依赖编译器特定细节，但下面是一种相对可移植的定义方式：

```cpp
#define _ASSERT_GLUE(a, b)  a ## b
#define ASSERT_GLUE(a, b)   _ASSERT_GLUE(a, b)

#define STATIC_ASSERT(expr) \
    enum \
    { \
        ASSERT_GLUE(g_assert_fail_, __LINE__) \
            = 1 / (int)(!!(expr)) \
    }

STATIC_ASSERT(sizeof(int) == 4);   // should pass
STATIC_ASSERT(sizeof(float) == 1); // should fail
```

其工作原理是定义一个包含单个枚举器的匿名枚举。枚举器的名称通过“粘合”一个固定前缀（如 `g_assert_fail_`）和一个唯一后缀生成，在本例中这个唯一后缀就是调用 `STATIC_ASSERT()` 宏所在的行号。枚举器的值被设置为 `1 / (!!(expr))`。双重否定 `!!` 确保 `expr` 具有布尔值。这个值随后被转换为 `int`，根据表达式是真还是假，分别得到 `1` 或 `0`。如果表达式为真，枚举器会被设置为值 `1/1`，也就是一。但如果表达式为假，我们就在要求编译器把枚举器设置为非法的 `1/0`，这会触发编译错误。

当上面定义的 `STATIC_ASSERT()` 宏失败时，Visual Studio 2015 会产生类似下面这样的编译期错误消息：

```text
1>test.cpp(48): error C2131: expression did not evaluate to
                 a constant
1>  test.cpp(48): note: failure was caused by an undefined
                  arithmetic operation
```

下面是另一种使用模板特化定义 `STATIC_ASSERT()` 的方式。在这个例子中，我们首先检查是否正在使用 C++11 或更高版本。如果是，就使用标准库实现的 `static_assert()`，以获得最大可移植性。否则，就回退到我们的自定义实现。

```cpp
#ifdef __cplusplus
    #if __cplusplus >= 201103L
        #define STATIC_ASSERT(expr) \
            static_assert(expr, \
                "static assert failed:" \
                #expr)
    #else
        // declare a template but only define the
        // true case (via specialization)
        template<bool> class TStaticAssert;
        template<> class TStaticAssert<true> {};

        #define STATIC_ASSERT(expr) \
            enum \
            { \
                ASSERT_GLUE(g_assert_fail_, __LINE__) \
                    = sizeof(TStaticAssert<!!(expr)>) \
            }
    #endif
#endif
```

这种使用模板特化的实现，可能比前面使用除以零的实现更可取，因为它在 Visual Studio 2015 中会生成稍微更好的错误消息：

```text
1>test.cpp(48): error C2027: use of undefined type
                 'TStaticAssert<false>'
1>  test.cpp(48): note: see declaration of
                  'TStaticAssert<false>'
```

不过，每个编译器处理错误报告的方式都不同，所以具体效果可能因环境而异。关于编译期断言的更多实现思路，一个不错的参考是 [119]。
