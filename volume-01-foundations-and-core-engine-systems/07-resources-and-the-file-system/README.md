# 7 资源与文件系统

游戏本质上是一种多媒体体验。因此，游戏引擎必须能够加载和管理大量不同类型的媒体资源——纹理位图、3D 网格数据、动画、音频片段、碰撞和物理数据、游戏世界布局，等等。此外，由于内存通常十分紧张，游戏引擎还必须确保任意时刻每个媒体文件在内存中只加载一份副本。例如，如果五个网格共用同一张纹理，那么我们希望内存中只有这一张纹理的副本，而不是五份。大多数游戏引擎都会使用某种形式的**资源管理器**（resource manager，也称 **asset manager** 或 **media manager**）来加载和管理构成现代 3D 游戏的各种资源。

每个资源管理器都会大量使用文件系统。在个人计算机上，文件系统通过一组操作系统调用库暴露给程序员。然而，游戏引擎通常会把原生文件系统 API “包装”成一个引擎专用 API，主要有两个原因。第一，引擎可能是跨平台的，在这种情况下，游戏引擎的文件系统 API 可以屏蔽不同目标硬件平台之间的差异，使软件的其他部分不必直接面对这些差异。第二，操作系统的文件系统 API 可能并不提供游戏引擎所需的全部工具。例如，许多引擎支持文件**流式加载**（file streaming，即游戏运行时“即时”加载数据的能力），但大多数操作系统并不会开箱即用地提供流式文件系统 API。主机游戏引擎还需要提供对多种可移动和不可移动介质的访问能力，从记忆棒、可选硬盘、Blu-ray 固定光盘，到网络文件系统（例如 Xbox Live 或 PlayStation Network，PSN）。不同类型介质之间的差异，同样可以被“隐藏”在游戏引擎的文件系统 API 之后。

在本章中，我们将首先探讨现代 3D 游戏引擎中常见的文件系统 API 类型。然后，我们会了解一个典型的资源管理器是如何工作的。
