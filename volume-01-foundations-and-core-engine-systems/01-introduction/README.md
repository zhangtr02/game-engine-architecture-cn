# 1 引言

当我在 1979 年拿到自己的第一台游戏主机——Mattel 出品、非常酷的 Intellivision 系统——时，“游戏引擎”这个术语还并不存在。在那个年代，大多数成年人都认为电子游戏和街机游戏不过是玩具；而驱动这些游戏运行的软件，也都高度专门化，既针对具体游戏本身，也针对其运行的硬件平台。如今，游戏已经成为一个价值数十亿美元的主流产业，其规模和受欢迎程度足以与好莱坞相媲美。而驱动这些如今无处不在的三维世界的软件，也就是像 Epic Games 的 Unreal Engine 5、Valve 的 Source 2 引擎、Crytek 的 CRYENGINE® 3、Electronic Arts DICE 的 Frostbite™ 引擎以及 Unity 游戏引擎这样的游戏引擎，已经发展为功能完备、可复用的软件开发套件，可以被授权并用于构建几乎任何可以想象的游戏。

虽然不同游戏引擎在架构与实现细节上差异很大，但无论是公开授权的商业游戏引擎，还是各公司内部专有的自研引擎，都已经呈现出一些可识别的粗粒度模式。几乎所有游戏引擎都包含一组熟悉的核心组件，包括渲染引擎、碰撞与物理引擎、动画系统、音频系统、游戏世界对象模型、人工智能系统，等等。在这些组件内部，通常也会采用相对少量的半标准化设计方案。

有很多书会对某个单独的游戏引擎子系统进行详尽介绍，例如三维图形学、碰撞检测或 AI。另一些书则会汇集游戏技术各个领域中的有用技巧和经验。然而，我一直没有找到一本书，能够为读者提供一幅相当完整的图景，系统展示构成现代游戏引擎的全部组件。因此，本书的目标，就是带领读者以一种有指导性、偏实践的方式，游览游戏引擎架构这一广阔而复杂的领域。

在本书中，你将学习：

- 真正工业级生产游戏引擎是如何进行架构设计的；

- 现实世界中的游戏开发团队是如何组织并开展工作的；

- 哪些主要子系统和设计模式会在几乎每一个游戏引擎中反复出现；

- 每个主要子系统通常具有哪些需求；

- 哪些子系统与游戏类型或具体游戏无关，哪些子系统通常会针对某一特定游戏类型或某个具体游戏进行显式设计；

- 引擎通常在哪里结束，而游戏又从哪里开始。

我们还将第一手了解一些流行游戏引擎的内部工作机制，例如 Quake、Unreal 和 Unity；同时也会了解一些知名中间件包，例如 Havok Physics 库、OGRE 渲染引擎，以及 RAD Game Tools 的 Bink 2 视频库。我们还将探索一些我有幸参与开发的专有游戏引擎，其中包括 Naughty Dog 为 *Uncharted* 和 *The Last of Us* 系列开发的引擎。

在正式开始之前，我们会先回顾一些在游戏引擎语境下进行大规模软件工程开发所需的技术与工具，包括：

- 逻辑软件架构与物理软件架构之间的区别；

- 配置管理、版本控制和构建系统；

- 针对 C 和 C++ 的常见开发环境之一 Microsoft Visual Studio 的一些使用技巧和经验。

在本书中，我假定你已经扎实理解 C++（大多数现代游戏开发者的首选语言），并且理解基本的软件工程原则。我也假定你已经接触过线性代数、三维向量与矩阵数学，以及三角学（不过我们会在第 5 章中回顾其中的核心概念）。理想情况下，你还应当对实时编程和事件驱动编程的基本概念有所了解。不过不用担心——我会简要回顾这些主题；如果你觉得在正式出发之前还需要进一步打磨相关技能，我也会为你指出正确的学习方向。
