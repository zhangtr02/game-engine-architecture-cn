## 9.2 与 HID 交互

所有人机接口设备都会向游戏软件提供输入，其中一些设备还允许软件通过各种输出向人类玩家提供反馈。游戏软件读取和写入 HID 输入与输出的方式多种多样，具体取决于目标设备的特定设计。

### 9.2.1 轮询

一些简单设备，例如游戏手柄和老式摇杆，是通过周期性**轮询**（polling）硬件来读取的（通常在主游戏循环的每次迭代中轮询一次）。这意味着显式查询设备状态；查询方式可以是直接读取硬件寄存器、读取内存映射 I/O 端口，也可以是通过更高层的软件接口完成（该接口内部再去读取相应的寄存器或内存映射 I/O 端口）。类似地，可以通过写入特殊寄存器或内存映射 I/O 地址，或者通过替我们完成底层工作的高层 API，将输出发送到 HID。

Microsoft 的 XInput API 可用于 Xbox 和 Windows PC 平台上的 Xbox 手柄，它就是一种简单轮询机制的很好例子。每一帧，游戏都会调用函数 `XInputGetState()`。该函数与硬件和/或驱动程序通信，以适当方式读取数据，并将其打包成方便软件使用的形式。它返回一个指向 `XINPUT_STATE` 结构体的指针，而该结构体又包含一个嵌入的 `XINPUT_GAMEPAD` 结构体实例。这个结构体包含设备上所有控件（按钮、拇指摇杆和扳机）的当前状态。

### 9.2.2 中断

有些 HID 只有在控制器状态以某种方式发生变化时，才会向游戏引擎发送数据。例如，鼠标会花很多时间静止不动地放在鼠标垫上。当鼠标没有移动时，没有理由在鼠标和计算机之间连续发送数据流；我们只需要在它移动、或者某个按钮被按下或释放时传输信息。

这类设备通常通过**硬件中断**（hardware interrupts）与主机计算机通信。中断是由硬件产生的电子信号，它会使 CPU 暂时暂停主程序的执行，并运行一小段称为**中断服务例程**（interrupt service routine, ISR）的代码。中断可以用于各种事情，但对于 HID 来说，ISR 代码很可能会读取设备状态，将其保存起来供稍后处理，然后把 CPU 交还给主程序。游戏引擎可以在下一次方便的时候取用这些数据。

### 9.2.3 无线设备

蓝牙设备（例如 Wiimote、PlayStation 的 DualShock 和 DualSense 控制器，以及 Xbox 无线控制器）的输入和输出，不能通过简单访问寄存器或内存映射 I/O 端口来读写。相反，软件必须通过蓝牙协议与设备“交谈”。软件可以请求 HID 将输入数据（例如按钮状态）发送回主机，也可以将输出数据（例如震动设置或一段音频数据流）发送到设备。这种通信通常由一个独立于游戏引擎主循环的线程处理，或者至少封装在一个相对简单、可从主循环调用的接口之后。因此，从游戏程序员的角度来看，蓝牙设备的状态可以被做得几乎与传统轮询设备没有区别。
